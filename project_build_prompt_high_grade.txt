Domain model (minimum):
1. Customer:
- customerId, name, contact, address, createdDate, createdAt, updatedAt
2. Vehicle:
- vehicleId, customerId, registrationNo(unique), make, model, year, createdAt, updatedAt
3. InsurancePolicy:
- policyId, customerId, vehicleId, startDate, endDate, premium, status, createdAt, updatedAt
- status enum: ACTIVE, EXPIRED, CANCELLED
4. Claim:
- claimId, policyId, vehicleId, accidentDate, claimDate, status, surveyorId(optional), workshopId(optional), damageDescription, createdAt, updatedAt
- status enum: SUBMITTED, UNDER_REVIEW, INSPECTED, APPROVED, REJECTED, WORKSHOP_ASSIGNED, CLOSED
5. InspectionReport:
- inspectionId, claimId, surveyorId, inspectionDate, estimatedCost, recommendation, remarks, createdAt, updatedAt
6. Workshop:
- workshopId, name, address, contact, status, createdAt, updatedAt
- status enum: REGISTERED, SUSPENDED
7. Staff:
- staffId, name, role, active
- role enum: SALESMAN, SURVEYOR, MANAGER

Business rules (strict):
1. Policy issuance requires valid existing customer and owned vehicle.
2. Vehicle registration number must be unique.
3. Claim filing requires ACTIVE policy and valid date window.
4. Claim can transition only through valid state machine rules.
5. Surveyor report submission requires assigned surveyor and proper claim state.
6. Manager approval/rejection requires existing inspection report.
7. Workshop assignment allowed only after claim APPROVED.
8. Workshop assignment allowed only if workshop status is REGISTERED.
9. Every state-changing operation logs an audit event.
10. Previous customer claims must be queryable efficiently from file data.

Persistence and reliability:
1. Repository interfaces for each aggregate.
2. File implementations using robust serialization (CSV with escaping OR JSON-like custom format).
3. Atomic write strategy:
- write to temp
- flush
- replace target
4. Corruption handling:
- detect malformed records
- skip with warning log
- continue service where possible
5. ID generation strategy that survives restarts.
